Introduction

This proposal adds SIMD types and operations to Javascript, focusing on the value semantics. The proposal adds new primitive types float32x4, etc, together with wrappers and a definition of their behavior in the language. The proposal discusses float32x4 and just a couple functions on it, with the idea that it would extend similarly to other SIMD types. Rough definitions for more types and functions that operate on them can be found at the SIMD.js spec. This document is organized in terms of where changes would be made to the ES6 spec.

The key problem that this spec aims to solve, over the previous informal SIMD specification, is the way that SIMD values were previously objects with identity-based equality. Maintaining object identity puts a big burden on compilers to maintain this identity through operations, where they would rather be able to duplicate and de-duplicate SIMD values arbitrarily based on algebraic identities. By making SIMD values into primitive types with structural equality, compilers are given more freedom. However, this gives a lot to expand on in the spec.

Ideally, SIMD values will fit into a larger value types proposal. Such a proposal would be a bit more involved, but good work has already been done in that direction. This document describes SIMD without a larger value type system, but it aims to be consistent with how value types might work, and once someone steps forward to describe value types in more detail, it will be great to simplify this text by just explaining SIMD in terms of value types.

1.26.1.8 SIMD types

1.2.16.1.8.1 Float32x4

Float32x4 is a SIMD type representing four 32-bit floating point values. Float32x4 values can be created using the [[Call]] operation on the SIMD.Float32x4 object, described in 27.1.1.

7.1 Type Conversion

1.37.1.1 ToPrimitive ( input [, PreferredType] )

Argument type: Float32x4
Result: return input Note An additional option considered is to define this as calling ToObject on it, and then ToPrimitive on the resulting wrapper object. This would have the advantage of allowing toString and valueOf be defined on the wrapper prototype, which seems more realistic to me as a path for value types.

1.47.1.2 ToBoolean ( argument )

Argument type: Float32x4
Result: return true Note Even if it would be consistent to return false if it is all zero, there probably isn't much gain from adding more falsy values.

1.57.1.3 ToNumber ( argument )

Argument type: Float32x4
Result: throw a TypeError exception

1.67.1.12 ToString ( argument )

Argument type: Float32x4
Result: Return "SIMD.Float32x4(" + ToString(ExtractLane(argument, 0)) + ", " + ToString(ExtractLane(argument, 1)) + ", " + ToString(ExtractLane(argument, 2)) + ", " + ToString(ExtractLane(argument, 3)) + ")" NoteAn alternative would be to call ToObject and let the wrapper handle ToString the way it works for objects.

1.77.1.13 ToObject ( argument )

Argument type: Float32x4
Result: Return a new %SIMDFloat32x4% object whose [[Float32x4Data]] internal slot is set to argument. See 27.1 for a description of %SIMDFloat32x4% objects.

1.87.2.1 RequireObjectCoercible ( argument )

Argument type: Float32x4

1.97.2.9 SameValue(x, y)

Add an extra step at the bottom of the definition of SameValue for the new case involving the new type:
  1. If Type(x) is Float32x4, then
    1. If SameValue(x.[[Float32x4X]], y.[[Float32x4X]]) and SameValue(x.[[Float32x4Y]], y.[[Float32x4Y]]) and SameValue(x.[[Float32x4Z]], y.[[Float32x4Z]]) and SameValue(x.[[Float32x4W]], y.[[Float32x4W]]), return true, otherwise return false

1.107.2.10 SameValueZero(x, y)

Add an extra step at the bottom of the definition of SameValue for the new case involving the new type:
  1. If Type(x) is Float32x4, then
    1. If SameValueZero(x.[[Float32x4X]], y.[[Float32x4X]]) and SameValueZero(x.[[Float32x4Y]], y.[[Float32x4Y]]) and SameValueZero(x.[[Float32x4Z]], y.[[Float32x4Z]]) and SameValueZero(x.[[Float32x4W]], y.[[Float32x4W]]), return true, otherwise return false

1.127.2.13 Strict Equality Comparison

Add a new step 9:
  1. If Type(x) is Float32x4, then
    1. If x.[[Float32x4X]] === y.[[Float32x4X]] and x.[[Float32x4Y]] === y.[[Float32x4Y]] and x.[[Float32x4Z]] === y.[[Float32x4Z]] and x.[[Float32x4W]] === y.[[Float32x4W]], return true
    2. Else, return false
